# 1. 项目概述

## 1.1 项目定位

企业流程审批管理系统是一套面向企业内部的流程管理与审批平台。

| 维度 | 说明 |
|------|------|
| 产品定位 | 以流程驱动项目的业务管理平台（非传统 OA 请假系统） |
| 核心理念 | 每一个流程 = 一个业务项目，每一个节点 = 一个完整工作单元 |
| 目标用户 | 系统管理员、所长（管理者）、普通员工 |

## 1.2 项目原则

本项目坚持以下原则：

1. 前后端分离
2. 模块化开发
3. 低耦合、高内聚
4. 流程模板与流程实例分离
5. 一个节点就是一个完整工作单元
6. 所有业务均可追溯
7. 文件全流程归档
8. 优先满足实际业务，再考虑功能扩展

## 1.3 系统边界

```text
企业流程审批系统
├── Dashboard（首页看板）    → 全局运行概况，所有角色可见
├── 流程管理（核心模块）      → 模板管理 + 实例管理 + 流程设计器
├── 个人中心（业务操作入口）  → 待办/审批/校验/发起流程/流程记录
└── 系统管理（基础数据维护）  → 用户/组织/角色/系统配置
```

> 一级菜单无二级菜单。个人中心仅所长和普通用户可见（系统管理员不参与业务，无个人中心）。系统管理内部用 Tab 切换。

## 1.4 设计原则

| 原则 | 说明 |
|------|------|
| 模板与实例分离 | 流程模板定义结构，实例独立运行，互不干扰 |
| 统一节点模型 | 不区分开始/工作/分支/结束节点，行为由位置决定 |
| 配置快照解耦 | 发起实例时生成完整配置快照，与模板后续修改解耦 |
| 状态驱动 | 所有业务对象通过状态字段驱动流转 |
| 日志不可删 | 所有操作记录日志，永久保留，不可删除 |
| 文件属实例 | 文件属于流程实例，按实例名称组织存储 |

# 2. 核心概念

## 2.1 统一节点模型

系统采用**统一节点模型**：所有节点均为同一种类型（Work Node），不区分开始/工作/分支/结束。节点的行为由其**在流程中的位置**决定。

| 位置 | 行为 | 是否可配置 | Task 生成 |
|------|------|:--:|:--:|
| 第一个（开始） | 系统默认生成，显示发起人姓名。发起后自动跳过 | 不可配置 | 不生成 Task |
| 中间（工作） | 负责人执行 → 上传文件 → 自动转 PDF → 提交 → 校验人并行校验 → 全部通过 → 审批人并行审批 → 全部通过后签名上 PDF → 下一节点 | 可配置 | 生成 Task |
| 最后一个（结束） | 发起人终审：查看全部文件 → 通过则归档；驳回则选择目标节点 | 不可配置 | 不生成 Task |

### 开始节点的行为
- 新建模板时自动生成，`is_start=1`
- 显示发起人姓名，无需配置
- 发起后自动跳过，直接进入第一个工作节点
- 不生成 Task，不需要审批

### 中间节点的行为
- 所长在流程设计器中配置：名称、负责人、校验人、审批人、时限、文件规则
- 支持并行（分叉/汇合）：一个节点可同时激活多个下游节点，全部完成后汇合
- 校验人在审批之前并行校验，全部通过后进入审批
- 审批人并行审批，全部通过后签名上 PDF，进入下一节点

### 结束节点的行为
- 新建模板时自动生成，`is_end=1`
- 审批人 = 流程发起人（终审）
- 发起人在「我的审批」中看到终审任务
- 通过 → 流程完成 → 归档至 `storage/archive/{实例名称}/`
- 驳回 → 选择目标节点 → 返回重新执行

## 2.2 节点标准流程

每个中间工作节点的标准执行流程：

```text
负责人收到 Task
  → 上传文件（Word/Excel/图片/PDF）
  → 提交时系统自动将非 PDF 转为 PDF（LibreOffice 无头模式）
  → 所有校验人并行校验
  → 全部校验通过 → 所有审批人并行收到审批
  → 全部审批通过（All Approve）→ 签名自动插入 PDF 固定位置（pypdf 库）
  → 进入下一节点
```

## 2.3 流程生命周期

```text
Created → Running → Completed
    ↓         ↓           ↓
    └─────────┴───────────┘
       Terminated（发起人取消，文件删除，不可恢复）

归档独立记录：archive_status = not_archived → archived；V1 终审通过时同步完成并归档。
```

| 状态 | 说明 |
|------|------|
| Created | 瞬时内部状态，实例创建后立刻推进到 Running，用户不可见 |
| Running | 流程运行中，节点按序推进 |
| Completed | 终审通过，流程完成 |
| 归档状态 | 不属于主状态；`archive_status` 为 `not_archived/archived`，归档时记录 `archived_at` |
| Terminated | 发起人主动终止，文件物理删除，数据库记录保留，不可恢复 |

## 2.4 核心对象链

```text
Organization → User
                   ↓
Flow Template → Flow Version → Flow Instance（含 priority 优先级属性）
                                        ↓
                                  Work Node（instance_nodes）
                                        ↓
                              Task → CheckRecord → Approval
                                        ↓
                                      File
                                        ↓
                                 Operation Log
```

| 对象 | 说明 |
|------|------|
| Organization | 组织（所），企业中的部门/团队 |
| User | 用户（含角色：system_admin / manager / user） |
| Flow Template | 流程模板，定义流程结构 |
| Flow Version | 模板发布后的版本快照（nodes + edges JSON） |
| Flow Instance | 模板发起后的运行实例，拥有优先级属性 |
| Work Node | 流程中的节点（instance_nodes） |
| Task | 中间节点自动生成的待办任务 |
| CheckRecord | 校验记录（校验人对提交内容的质量门禁检查） |
| Approval | 审批记录 |
| File | 文件记录（业务归属实例，并通过 node_id/task_id/round 定位产生节点、任务和轮次） |
| Operation Log | 操作日志（只写不删） |

# 3. 系统架构

## 3.1 技术栈

| 层 | 技术 | 说明 |
|----|------|------|
| 前端框架 | Vue 3 + TypeScript | Composition API |
| 流程设计器 | LogicFlow | 滴滴开源，专为审批流场景设计 |
| UI 组件库 | Element Plus | 企业级组件库 |
| 后端 | FastAPI (Python 3.10+) | 异步框架，自动生成 Swagger 文档 |
| ORM | SQLAlchemy 2.0 | 异步模式 |
| 数据库 | MySQL 8.0 | InnoDB 引擎，操作日志表按年分区 |
| 文件存储 | 服务器本地目录 | `storage/archive/{实例名称}/` |
| 认证 | JWT | python-jose 库 |
| PDF 转换 | LibreOffice 无头模式 + Pillow | asyncio.Semaphore 限流 2 并发 |
| PDF 签名 | pypdf | 签名图片插入 PDF 指定坐标 |
| 异步任务 | FastAPI BackgroundTasks | V1 不引入 Celery |

## 3.2 角色体系（RBAC）

| 角色 | 标识 | 核心职责 |
|------|------|----------|
| 系统管理员 | system_admin | 用户/组织/角色维护，系统配置管理。**不参与业务流程** |
| 所长 | manager | 设计流程模板、发起流程实例、终止流程、终审 |
| 普通用户 | user | 执行节点任务、上传文件、校验、审批 |

## 3.3 模块结构

```text
企业流程审批系统
├── Dashboard（首页看板）
├── 流程管理（内部页面导航：全部流程 / 组织卡片 / 实例列表 / 模板列表）
│   └── 流程设计器（LogicFlow 可视化设计）
├── 个人中心（卡片分区，一页展示；仅所长和普通用户可见）
└── 系统管理（Tab 切换：用户/组织/角色/系统配置；仅管理员可见）
```

## 4.2 权限规则

- 所有已登录用户均可查看所有组织的流程模板和实例
- 仅流程所属组织的所长可编辑该流程模板
- 仅节点负责人可执行该节点
- 仅配置的审批人可审批该节点
- 负责人和审批人可为同一人，系统不拦截
- 校验人与负责人不建议为同一人（系统不强制拦截）
- 跨所协作：负责人和审批人可跨组织，但模板编辑权限属于所属所
- 前端根据角色隐藏无权限按钮和菜单，真正权限校验统一由后端完成

# 5. 功能模块

## 5.1 Dashboard（首页看板）

Dashboard 是用户登录后的首页，展示系统全局运行概况。所有角色看到的内容完全一致。

### 四大统计卡片

| 卡片 | 统计内容 | 说明 |
|------|----------|------|
| 进行中项目 | 全系统 Running 状态的实例总数 | 系统活跃度 |
| 已归档项目 | 全系统 `archive_status=archived` 的实例总数 | 累计成果 |
| 本月归档 | 本月 `archived_at` 非空且落在本月的实例数 | 团队吞吐量 |
| 超期预警 | 全系统已逾期 + 即将逾期的 Task 总数 | 风险预警 |

### 任务状态分布（饼图）

展示全系统所有 Task 的当前状态分布（不含开始/结束节点）：

| 状态 | 说明 |
|------|------|
| Pending（待处理） | 负责人尚未开始处理 |
| Processing（处理中） | 负责人已开始但未提交 |
| Waiting Check（待校验） | 已提交，等待校验中 |
| Waiting Approval（待审批） | 已通过校验，等待审批中 |
| Completed（已完成） | 近 30 天内完成的 Task |

### 流程卡点追踪（表格）

展示所有运行中流程实例的完整进度 —— 全部工作节点以标签链展示：

```text
[✅资料上传 张三] → [✅设计评审 李四] → [🔵毛坯图下发 王五] → [⚪全套图纸] → [⚪BOM下发]
```

- 只显示工作节点（不含开始/结束节点）
- 标签状态：✅已完成 / 🔵进行中 / ⚪待开始
- 排序：优先级优先（紧急→高→普通→低），同优先级内按逾期→即将逾期→正常

### 超期预警列表

展示全系统中已逾期和即将逾期的任务：

| 列 | 说明 |
|----|------|
| 流程实例名称 | 所属流程 |
| 当前节点 | 逾期节点名称 |
| 负责人 | 节点负责人 |
| 截止时间 | 节点完成时限 |
| 剩余天数 | 还剩 X 天 / 已逾期 X 天 |
| 所属组织 | 归属的所 |

### 各所流程概览（折叠面板）

按组织分组，展示每个所下正在运行的流程实例。默认展开所有组织。

## 5.2 流程管理

流程管理是所长创建和维护流程模板的核心入口。所有用户均可查看，但仅所长可在本所下编辑。

### 组织卡片页

进入流程管理时，以卡片形式展示所有组织概览：

- 「全部流程」卡片：全系统实例总数，点击进入全局实例列表
- 各组织卡片：组织名称、运行中实例数、最近更新时间、当前用户所属组织高亮

### 流程实例 Tab（默认选中）

日常高频操作入口，展示所选范围的流程实例：

| 列 | 说明 |
|----|------|
| 实例名称 | 发起时填写的名称 |
| 模板来源 | 使用哪个流程模板 |
| 优先级 | 紧急/高/普通/低（彩色标签） |
| 所属组织 | 全部流程视图显示，单所视图隐藏 |
| 当前负责人 | 当前节点负责人 |
| 节点进度 | 当前第 N/M 个节点 |
| 状态 | 运行中/已完成/已终止 |
| 发起时间 | 实例创建时间 |

- 支持 keyword 模糊搜索实例名称
- 状态筛选：全部/运行中/已完成/已终止
- 默认显示「运行中」

### 流程模板 Tab

展示模板列表，所长可创建/编辑/管理模板：

| 列 | 说明 |
|----|------|
| 流程名称 | 流程模板名称 |
| 当前版本 | 最新发布的版本号 |
| 状态 | 草稿/已发布/已停用 |
| 节点数 | 模板中包含的节点数量 |
| 更新时间 | 最后修改时间 |

模板操作权限：

| 操作 | 草稿 | 已发布 | 已停用 | 权限条件 |
|------|:--:|:--:|:--:|------|
| 查看 | YES | YES | YES | 全部用户 |
| 编辑 | YES | YES（仅软修改） | NO | 本所所长 |
| 发布 | YES | NO | NO | 本所所长 |
| 停用 | NO | YES | NO | 本所所长 |
| 删除 | YES | NO | NO | 本所所长 |
| 发起流程 | NO | YES | NO | 本所所长 |

### 流程模板创建

创建模板时自动生成开始节点（is_start=1）和结束节点（is_end=1），默认名称为"开始"和"结束"。开始节点显示发起人姓名，结束节点审批人=发起人。

### 发布校验

发布前系统必须校验以下内容：

| 校验项 | 规则 |
|--------|------|
| 流程名称 | 必填，2-50 字符 |
| 节点数量 | 至少包含 3 个节点（开始 + 至少1个工作节点 + 结束） |
| 节点连线 | 所有节点必须连通（无孤立节点）。从开始节点 BFS 遍历，检查所有非开始/结束节点是否都可达，所有路径是否都可到达结束节点。分叉/汇合连线合法 |
| 分叉/汇合 | fork 节点可连接多个下游，join 节点可被多个上游连接。禁止自环（节点连接到自身） |
| 节点名称 | 每个工作节点名称必填 |
| 节点负责人 | 每个工作节点必须指定负责人 |
| 节点校验人 | 每个工作节点至少指定一个校验人 |
| 节点审批人 | 每个工作节点至少指定一个审批人 |
| 节点时限 | 每个工作节点必须设置完成时限 |

### 发起流程实例

1. 选择已发布模板 → 填写实例名称
2. （可选）调整节点配置：更换负责人/校验人/审批人、调整截止日期、跳过可选节点
3. 选择优先级（urgent/high/normal/low，默认 normal）
4. 确认发起 → 生成配置快照 → 与模板解耦

不可自定义项：节点数量、节点顺序、连线关系（属于模板结构）。

## 5.3 流程设计器（Flow Designer）

基于 LogicFlow 的可视化流程设计器，是系统核心功能模块。

### 设计原则

- 统一节点模型：所有节点使用相同矩形样式
- 首个节点（开始）和最后一个节点（结束）由系统默认生成，不可删除
- 中间节点支持串联和并行（fork/join）两种结构
- V1 不实现条件分支，但支持 fork/join 并行分叉汇合

### 节点属性配置

选中中间节点后，在属性面板中配置：

| 属性 | 类型 | 必填 | 说明 |
|------|------|:--:|------|
| 节点名称 | 文本输入 | YES | 如"资料上传"，2-30 字符 |
| 节点描述 | 文本域 | NO | 描述本节点工作内容，最多 500 字符 |
| 负责人 | 用户搜索选择 | YES | 单选，支持跨组织 |
| 校验人 | 用户多选 | YES | 至少 1 人，支持跨组织 |
| 完成时限 | 数字输入 | YES | 单位：天，正整数（模板用天数） |
| 必须上传文件 | 开关 | NO | 开启后提交节点前必须上传至少一个文件 |
| 审批人 | 用户多选 | YES | 至少 1 人，支持跨组织 |
| 审批策略 | 单选 | 锁定 | V1 固定"全部通过"，不可修改（预留扩展） |
| 可选节点 | 开关 | NO | 默认关闭。开启后发起实例时可选择跳过此节点 |

> 开始节点（is_start=1）和结束节点（is_end=1）不可配置，选中时面板显示"系统默认节点，无需配置"。

### 画布操作

- **添加节点**：点击或拖拽添加工作节点，默认名"新节点"
- **删除节点**：选中后 Delete，开始/结束节点不可删除
- **连线**：从输出端口拖拽至输入端口。分叉（fork）可连接多个下游，汇合（join）可被多个上游连接。禁止自环
- **缩放平移**：鼠标滚轮缩放（10%-200%），拖拽画布空白区域平移
- **撤销重做**：Ctrl+Z/Ctrl+Y，上限 50 步

### 节点视觉状态

- 配置完整的节点：实线边框，显示名称/负责人/时限/审批人
- 未配置节点：橙色虚线边框，右上角红色圆点标记，显示"未配置"/"未设置"
- 校验人和审批人每行最多显示 2 人，超出显示"+N"

### 节点列表视图

画布下方提供表格视图，以列表形式展示所有节点的配置，方便节点数较多时快速核对。

### 保存与发布

- **保存草稿**：不做校验，保存当前全部内容
- **发布**：执行完整校验 → 通过后生成版本，状态变为 published
- 已发布模板支持软修改和硬修改。软修改写入当前版本的节点软配置覆盖层，仅覆盖 `description/assignee/time_limit/checkers/approvers`，只影响之后发起的新实例；已发起实例完全使用自身快照。`is_optional`、节点/连线及其他结构字段均属硬修改，自动生成新版本。

```text
┌─────────────────────────────────────────────┐

                      ↓

                      ↓

│                 → approvals                  │

```

|---|------|------|--------|

| 2 | users | 用户（含签名图片路径） | 基础数据 |

## 6.3 核心设计

### 模板与实例分离

- `flow_templates` 负责定义流程结构，`flow_instances` 负责运行
- 实例基于 `flow_versions` 快照创建，与模板后续修改解耦
- flow_versions 存储 `nodes_snapshot` 和 `edges_snapshot`（JSON），发布时一次性生成，不可修改

### 版本快照机制

```text
Draft ──发布──→ Published (V1.0)
                    │
                    ├── 软修改（审批人/时限/描述）→ 即时生效，版本号不变
                    │
                    └── 硬修改（节点/连线变更）→ 自动创建新 Draft → 发布 → V2.0
```

- 软修改：改审批人、改负责人、改时限、改描述。不产生新版本，即时生效
- 硬修改：增删节点、改动连线关系、改流程结构。自动版本号 +1，新发起的实例使用新版本
- 运行中实例不受模板修改影响

### 统一节点模型的数据表达

```text
template_nodes / instance_nodes:
  is_start = 1   → 开始节点
  is_start = 0, is_end = 0  → 中间工作节点
  is_end = 1     → 结束节点
```

中间节点的审批人/校验人列表使用 JSON 存储，避免过度多表关联：

```json
{
  "approvers": [
    {"user_id": 3, "name": "李四"},
    {"user_id": 5, "name": "王五"}
  ],
  "checkers": [
    {"user_id": 6, "name": "钱七"},
    {"user_id": 7, "name": "孙八"}
  ]
}
```

### 并行汇合控制

通过 `incoming_count` 和 `arrived_count` 字段实现：

- 发起实例时为每个节点计算 `incoming_count`：查询 `instance_edges WHERE target_node_id = ?` 统计上游连线数
- 串联节点 incoming_count = 1，汇合点 incoming_count > 1
- 每个节点完成或可选工作节点被跳过时，仅其直接下游节点的 `arrived_count` + 1；当前节点自身不增加。跳过信号沿原图继续传播，不修改或绕过边
- 当 `arrived_count == incoming_count` 时，所有上游分支均已到达，激活汇合节点

### 文件属于实例

- files 表关联 instance_id，不关联 user
- 文件存储路径：`storage/archive/{实例名称}/`
- 提交时先完成全部非 PDF 转换；成功后删除原件，仅以 `file_path` 保存最终 PDF，`mime_type=application/pdf`；失败不得进入校验

# 7. 节点与状态机

## 7.1 中间节点属性

| 属性 | 类型 | 必填 | 说明 |
|------|------|:--:|------|
| 节点名称 | VARCHAR(30) | YES | 如"资料上传" |
| 节点描述 | VARCHAR(500) | NO | 描述本节点工作内容 |
| 负责人 | FK→users | YES | 节点任务执行者 |
| 校验人 | JSON | YES | 至少 1 人，校验在审批之前 |
| 完成时限 | INT（天数） | YES | 模板阶段用天数，实例激活时计算为具体 deadline |
| 必须上传文件 | TINYINT(1) | NO | 默认 0 |
| 审批人 | JSON | YES | 至少 1 人 |
| 审批策略 | VARCHAR(30) | 锁定 | V1 固定 all_approve |
| 可选节点 | TINYINT(1) | NO | 默认 0，开启后发起实例时可跳过 |
| 被跳过标记 | TINYINT(1) | NO | is_skipped，发起时选择跳过后设为 1 |
| 轮次 | INT | — | round，被驳回重新进入时 +1 |
| 上游连线数 | INT | — | incoming_count，汇合控制 |
| 已到达数 | INT | — | arrived_count，汇合控制 |

## 7.2 instance_nodes 状态

```text
中间工作节点：
waiting → running → waiting_check → waiting_approval → finished
                       ↓                ↓
                running（校验/中间审批退回当前负责人）  rejected（仅终审总驳回触发跨节点回退）
                       ↓
                running（回到负责人修改）
                terminated（流程被终止）

开始节点：
waiting → finished（发起后自动完成）

结束节点：
waiting → waiting_approval（发起人终审）→ finished
              ↓
         rejected（驳回至指定节点）

可选节点被跳过：
waiting → skipped
```

## 7.3 tasks 状态

```text
pending → processing → waiting_check → waiting_approval → completed
                    ↓                ↓
             processing（校验/中间审批退回）  rejected（终审总驳回导致旧任务关闭）
             terminated（流程被终止）
```

> 开始节点和结束节点不生成 Task。

## 7.4 check_records 状态

```text
pending → passed（校验通过，等待其他校验人）
pending → returned（校验退回，Task 回到 processing）
pending → terminated（流程终止或紧急换人关闭待处理记录）
```

- 多个校验人并行校验，各自独立决定
- 全部 passed → 进入审批环节，生成 Approval 记录
- 任一 returned → Task 回到 processing，负责人重新提交，重新生成 CheckRecord

## 7.5 approvals 状态

```text
pending → approved（通过，签名自动上 PDF）
pending → rejected（驳回）
pending → terminated（流程被终止）
```

- 多个审批人并行审批，各自独立决定
- 全部 approved → 节点 finished，推进到下一节点
- 任一 rejected → 节点 rejected，回退至指定节点

## 7.6 完整状态流转图

```text
发起流程
  ↓
开始节点（自动跳过）→ finished
  ↓
工作节点1 → running → Task: pending → processing
  → 提交 → waiting_check → 校验(全部passed) → waiting_approval
  → 审批(全部approved) → finished → 签名上PDF → 下一节点
  ↓
工作节点2 → ...（循环）
  ↓
...
  ↓
结束节点 → waiting_approval → 发起人终审
  ├── 通过 → Completed，同时 archive_status → archived
  └── 总驳回 → 返回已执行中间工作节点 → 重新执行
```

# 8. 文件与存储

## 8.1 存储结构

```text
storage/
├── archive/                    ← 流程归档目录
│   └── {实例名称}/              ← 每个实例一个目录
│       ├── abc123.pdf          ← 原始文件
│       ├── def456.pdf          ← 转换后的 PDF
│       └── ...
└── signatures/                 ← 签名图片目录
    └── user_{user_id}.png      ← 用户签名
```

## 8.2 文件规则

| 规则 | 说明 |
|------|------|
| 文件归属 | 文件属于流程实例（关联 instance_id），不属于用户或节点 |
| 上传类型 | normal（正常流程文件）/ supplement（补交文件） |
| 提交时转换 | 所有非 PDF 文件自动转换为 PDF；成功后仅保留 PDF 并删除原文件，失败则清理临时原件且不得进入校验 |
| 转换并发 | asyncio.Semaphore 限流 2 并发 |
| 文件类型 | .doc/.docx/.xls/.xlsx/.pdf/.png/.jpg/.jpeg |
| 大小限制 | 单文件 ≤ 50MB |
| 驳回处理 | 驳回节点的文件物理删除，数据库记录同步删除 |
| 已提交文件 | 不可删除（不显示删除按钮） |
| 未提交文件 | 当前节点上传且未提交的文件可删除 |

## 8.3 PDF 签名机制

| 属性 | 说明 |
|------|------|
| 签名图片格式 | PNG，透明底，建议 200x60px，≤500KB |
| 存储路径 | `storage/signatures/user_{user_id}.png` |
| 插入方式 | 审批通过时，pypdf 将签名插入 PDF 指定坐标 |
| 签名位置 | 系统配置中设定（默认：最后一面右下角，X=400, Y=100） |
| 多签排列 | 按审批顺序水平排列，每个签名向右偏移 150px，超出页面宽度自动换行 |
| 未上传签名 | 可正常审批，审批通过时跳过签名插入步骤 |

## 8.4 补交文件（兜底机制）

- 除 terminated 外可按状态和身份补交文件
- 权限：running 为发起人和当前负责人；completed（无论是否已归档）为发起人和对应节点历史负责人；校验人/审批人身份本身不授权；terminated 禁止
- 自动转 PDF
- 文件记录标记 `upload_type = 'supplement'`
- 不影响流程状态和节点推进

# 9. 业务流程

## 9.1 发起流程

```text
所长登录 → 选择已发布模板 → 填写实例名称
  → （可选）选择优先级（紧急/高/普通/低）
  → （可选）展开节点配置，逐节点调整：
    - 更换负责人/校验人/审批人
    - 调整截止日期（日期选择器，默认 = 发起日期 + 模板天数）
    - 跳过可选节点（模板中标记为可选的节点）
  → 确认发起
  → 系统以调整后的配置生成 instance_nodes 快照（与模板解耦）
  → 开始节点自动跳过（finished）
  → 推进到第一个非跳过的工作节点 → 生成 Task → 计算 deadline
  → 实例状态 → running
  → 记录操作日志
```

## 9.2 节点执行流程

```text
节点激活 → Task 生成（pending）
  → 负责人打开 Task → 状态变为 processing
  → 上传文件（Word/Excel/图片/PDF）
  → （可选）保存草稿
  → 提交节点：
    1. 前置校验（若 require_file=true，检查是否有文件）
    2. 异步将所有非 PDF 文件转为 PDF
    3. Task → waiting_check
    4. instance_node → waiting_check
    5. 按 checkers 列表为每个校验人生成 CheckRecord（pending）
  → 所有校验人并行校验：
    - 全部 passed → 进入审批
    - 任一 returned → Task 回到 processing（不生成新 Task）
  → 全部校验通过：
    1. instance_node → waiting_approval
    2. Task → waiting_approval
    3. 按 approvers 列表为每个审批人生成 Approval（pending）
  → 所有审批人并行审批：
    - 全部 approved → 签名上 PDF → 节点 finished → 推进到下一节点
    - 任一 rejected → 固定退回当前负责人，不可选择跨节点目标
```

## 9.3 审批通过后的节点推进

```text
当前节点全部审批通过：
  1. instance_node → finished
  2. 遍历当前节点的所有直接下游节点（通过 instance_edges）
  3. 对每个下游节点：
    - arrived_count + 1
    - 若 arrived_count == incoming_count（汇合条件满足）：
      - 若下游是结束节点 → 直接创建 Approval（task_id=NULL, approver=发起人）
      - 若下游是中间节点 → 状态 → running，生成 Task，计算 deadline
    - 若 arrived_count < incoming_count → 继续等待其他分支
```

## 9.4 驳回流程

```text
中间节点审批人点击退回：
  1. 目标固定为当前节点负责人，不提供目标节点选择
  2. 填写驳回原因（必填）
  3. 确认后：
    - Approval → rejected
    - 当前 instance_node → rejected
    - 当前 Task → rejected
    - 当前轮文件和 files 记录立即物理删除
    - 当前节点回到 running，沿用当前 Task 回到 processing，round 不变
```

### 终审总驳回下游级联规则

仅结束节点发起人终审总驳回可选择任意已执行中间工作节点。目标节点当前文件立即删除；受影响下游回退为 waiting，并在重新执行到达时删除旧文件。开始/结束/未执行/跳过节点不可选。

### 并行流程驳回规则

- **驳回在分支内**：仅该分支及下游到汇合点之间的节点受影响，其他分支不受影响
- **驳回在分叉点或分叉点之前**：分叉点之后的所有分支全部回退

## 9.5 终审流程

```text
所有中间节点审批通过 → 结束节点自动进入 waiting_approval
  → 为发起人生成 Approval（task_id=NULL）
  → 发起人在「我的审批」中看到终审任务
  → 进入终审页面，可查看全部节点文件
  → 操作：
    - 终审通过 → 流程 Completed → 签名上 PDF → 归档
    - 总驳回 → 填写原因 → 从已执行中间工作节点选择目标 → 返回该节点重新执行
```

## 9.6 终止流程

```text
发起人在「我发起的流程」中点击终止：
  1. 填写终止原因（必填，≤500 字符）
  2. 二次确认
  3. 确认后系统自动：
    - 流程状态 → terminated（数据库记录保留）
    - 所有非终态 instance_nodes → terminated
    - 所有非终态 tasks → terminated
    - 所有 pending check_records / approvals → terminated
    - 物理删除实例全部文件及 files 记录
    - 操作记入日志
  4. 不可撤销，流程变为只读
```

> 任意未 terminated 实例均可终止，包括 completed 且已归档的实例。

## 9.7 紧急换人

```text
发起人在实例详情页 → 展开节点卡片 → 点击「修改人员」：
  1. 选择要修改的节点（仅未完成节点可修改）
  2. 更换负责人/校验人/审批人
  3. 确认后：
    - 更新 instance_nodes 中对应字段
    - 已有 CheckRecord 不在新列表且 pending → terminated
    - 已有 Approval 不在新列表且 pending → terminated
    - 已完成的校验/审批保留不动
    - 新校验人生成 CheckRecord（pending）
    - 新审批人生成 Approval（pending）
    - 若节点 running 且仅换负责人 → Task.assignee_id 同步更新
    - 记录操作日志（变更详情）
```

## 9.8 优先级修改

仅发起人可在实例详情页顶部基本信息区修改 `running` 实例的优先级（urgent/high/normal/low），操作记入日志。

# 10. 关键业务规则

以下 12 条规则是整个系统的核心约束，贯穿设计、开发和运行全生命周期：

1. **模板与实例分离**：一个流程模板有多个版本，一个版本可发起多个实例。模板修改不影响已运行实例。

2. **软修改与硬修改**：已发布模板的软修改（审批人/时限/描述）即时生效不产生新版本；硬修改（节点/连线变更）自动生成新版本。运行中实例不受影响。

3. **实例发起时逐节点可调**：发起时可逐节点调整审批人/负责人/截止日期，发起后生成配置快照，与模板解耦。

4. **开始和结束节点不可删除**：由系统默认生成，在流程设计器中不可删除、不可配置（第一个和最后一个节点）。

5. **发布校验**：至少 3 个节点（开始 + 至少 1 个工作节点 + 结束）；所有中间节点完整配置（名称/负责人/审批人/校验人/时限）；所有节点全部连通（BFS 校验）。

6. **跨所协作**：负责人、校验人和审批人可跨组织指定，但模板编辑权限属于模板所属所。

7. **操作日志不可删除**：所有操作记录在 operation_logs 表，仅 INSERT 和 SELECT，无 UPDATE/DELETE。日志表按年分区，长期保留。

8. **V1 范围限制**：第一版不做条件分支（支持 fork/join 并行分叉汇合），不做消息通知。审批策略固定"全部通过"。

9. **发起人终止权**：仅发起人可终止自己发起的流程，任意未 terminated 状态均可终止（Created/Running/Completed，含已归档）。终止后文件物理删除，files 记录同步删除，日志和实例元数据保留，不可恢复。

10. **审批自动签名**：用户上传签名图片后，审批通过时系统自动用 pypdf 将签名插入 PDF 指定位置。未上传签名的用户可正常审批，跳过签名步骤。

11. **紧急换人**：发起人可随时更换运行中实例未完成节点的负责人/校验人/审批人。待处理的校验/审批记录自动更新（pending → terminated），已完成的保留不动。

12. **优先级可选/可改**：发起实例时可选择优先级（urgent/high/normal/low），默认 normal。发起后仅发起人可修改 `running` 实例的优先级。

# 12. 设计原则

> 本章节定义整个系统的设计原则。所有开发人员、AI、后续维护人员都必须遵循本章节。本章节优先级仅次于 Blueprint。

---

## 12.1 核心设计思想

整个系统遵循以下设计思想：

> **业务驱动设计（Business Driven Design）**

系统不是围绕页面开发，不是围绕数据库开发，而是围绕业务流程开发。

因此：

```text
业务 → 流程 → 数据 → 接口 → 页面
```

而不是：

```text
页面 → 数据库 → 业务
```

设计时始终问：这个功能解决了什么业务问题？属于哪个流程？驱动哪个节点？

---

## 12.2 架构原则

整个系统采用 **前后端分离 + 分层架构 + 模块化开发**。

统一分层：

```text
Frontend → API → Service → Flow Engine → Repository → Database
```

各层职责独立，不得跨层调用。前端不直接操作数据库，后端不控制页面展示。

---

## 12.3 模块原则

每个模块只负责一件事情，模块之间通过 API 通信，禁止直接引用内部逻辑。

```text
Dashboard     → 展示全局运行数据
流程管理      → 管理流程模板和实例
个人中心      → 处理用户个人业务（任务、审批、校验、发起流程）
系统管理      → 维护基础数据（用户/组织/角色/配置）
```

流程设计器作为流程管理的子功能存在，Task 和 Approval 通过个人中心处理，不作为一级菜单。

---

## 12.4 Flow First 原则

整个系统遵循 **流程优先（Flow First）**。

所有业务必须依附流程。不能存在没有流程的文件上传，也不能存在脱离流程实例的审批。任何业务必须属于某一个流程实例。

```text
流程实例 → 节点 → 上传文件  ✓
上传文件 → 没有流程        ✗
```

---

## 12.5 单一数据源原则（SSOT）

系统遵循 **唯一数据源原则（Single Source of Truth）**：

- 流程状态只能来自 Flow Instance
- Task 状态只能来自 Task 表
- 审批状态只能来自 Approval 表
- 校验状态只能来自 CheckRecord 表

前端页面不得自行计算业务状态，所有状态以数据库为准。页面仅负责根据状态展示对应 UI。

---

## 12.6 状态驱动原则

整个系统采用状态驱动，而不是页面驱动。

```text
waiting → running → waiting_check → waiting_approval → finished
```

页面根据状态展示不同 UI，后端根据状态执行不同逻辑。任何业务不得跳过状态，所有流转必须按照状态机定义的方向进行。

---

## 12.7 数据一致性原则

业务数据必须保持一致。例如：

```text
Task Completed → Node Finished → Flow 推进到下一节点
```

三者状态必须同步。不得出现 Task 已完成但 Node 未完成的情况。关键状态变更必须在同一事务中完成（如审批通过时的行锁 + 签名 + 状态更新）。

---

## 12.8 文件管理原则

文件属于 Flow Instance，不属于 User、Task 或 Node。

原则：
- 每个节点可新增文件，后续节点可查看之前所有文件
- 提交时所有非 PDF 文件自动转换为 PDF
- 已提交文件不可删除，未提交文件可删除
- 驳回场景下旧文件物理删除、重新上传新文件
- 流程结束统一归档至 `storage/archive/{实例名称}/`
- 补交文件不影响流程状态和节点推进

---

## 12.9 日志原则

任何业务操作必须记录日志，包括：创建流程、修改流程、发布流程、发起流程、上传文件、提交任务、校验、审批、驳回、终止、补交文件、紧急换人、优先级修改。

日志不可修改、不可删除、永久保存。采用 MySQL 按年分区表（RANGE 分区 on created_at），建表时提前建好未来 10 年分区并按年新增分区；禁止通过 DELETE 或 DROP PARTITION 清理历史日志。自动操作使用 `operator_type=system`、`operator_id=NULL`，可用 `triggered_by` 追溯触发人。

---

## 12.10 权限原则

所有权限由后端控制。前端仅负责隐藏按钮和菜单项，真正权限判断统一由后端 API 层完成。任何接口均必须校验权限，不得依赖前端隐藏作为安全手段。

---

## 12.11 可扩展原则

第一版（MVP）仅实现核心业务，但所有模块必须预留扩展能力。

预留方向：
- 流程引擎：会签/或签/条件分支/子流程
- 权限：多角色/多组织/多租户
- 审批策略：任一通过/按比例通过
- 通知：WebSocket/邮件/企业微信
- AI：审批建议/流程设计助手/异常分析

不得为了第一版速度牺牲扩展能力。数据库设计、接口设计、前端组件均需考虑扩展场景。

### 日志测试

|------|:--:|------|
| 条件分支 | 高 | 支持 if/else 条件路由 |

## 14.4 远期版本展望

### V2.0 流程增强版本

增强流程能力：会签/或签/子流程/流程嵌套。目标：支持复杂企业流程。

### V3.0 企业协作版本

增强协作能力：消息通知、邮件通知、企业微信/钉钉接入、跨组织协作增强。目标：企业级协同办公能力。

### V4.0 平台化版本

系统升级为平台：多租户（SaaS 化）、流程市场、流程组件化、可拖拽流程插件系统、API 开放平台。目标：从系统走向平台。

### V5.0 智能化版本（AI 增强）

引入 AI 能力：AI 审批建议、AI 流程设计助手、自动节点优化、异常流程分析、流程瓶颈识别、自动生成流程模板。目标：AI 驱动流程系统。

---

## 14.5 长期愿景

最终系统形态：**企业 Workflow OS（流程操作系统）**

能力包括：所有业务流程化、所有流程可视化、所有任务可追踪、所有文件可流转、所有审批可审计、所有流程可优化、AI 参与流程设计与执行。

---

## 14.6 成功标准

### 技术层
- 稳定运行，支持复杂流程
- 状态机完整，流程引擎健壮
- 架构可扩展，模块可替换

### 业务层
- 一个流程可以完整跑通（设计 → 发起 → 执行 → 审批 → 归档）
- 多人协作无冲突（并行校验、并行审批）
- 文件全程可追溯，签名自动上 PDF
- 审批链完整，驳回规则明确

### 产品层
- 用户能理解流程进度
- 所长能独立设计流程模板
- 用户能高效执行节点任务
- 管理员可维护系统基础数据
