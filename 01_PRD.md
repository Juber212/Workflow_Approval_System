# 1. 产品概述

## 1.1 产品简介

企业流程审批系统是一套面向企业内部的流程管理与审批平台。系统以"流程（Flow）"为核心，将企业中的每一个业务项目抽象为一个流程实例，通过工作节点（Work Node）驱动业务执行，通过审批（Approval）控制流程推进，实现从项目立项到归档的全生命周期管理。

## 1.2 产品定位

​	是企业项目流程审批管理平台

- 每一个流程 = 一个业务项目
- 每一个节点 = 一个完整工作单元

## 1.3 目标用户

| 用户类型 | 角色 | 核心诉求 |
|----------|------|----------|
| 系统管理员 | System Admin | 维护系统基础数据，管理用户与组织 |
| 所长 | Manager | 设计流程、发起流程、管理本所业务 |
| 普通用户 | User | 执行任务、上传文件、审批节点 |

## 1.4 产品架构

```
企业流程审批系统
├── Dashboard（首页看板）
├── 流程管理（内部页面导航）
├── 个人中心（卡片分区，一页展示）
└── 系统管理（Tab 切换，仅管理员可见）
```

## 1.5 核心业务流程

```
创建流程模板 → 发布版本 → 发起流程实例 → 节点执行 → 校验 → 审批 → 下一节点 → ... → 流程结束 → 归档
```

# 2. 用户角色与权限

## 2.1 角色定义

系统共设三个角色：

| 角色 | 标识 | 说明 |
|------|------|------|
| 系统管理员 | system_admin | 系统维护者，不参与业务流程 |
| 所长 | manager | 组织管理者，负责流程设计与管理 |
| 普通用户 | user | 流程执行者与审批参与者 |

## 2.2 权限矩阵

> ✅ = 可操作 &nbsp;&nbsp; ❌ = 不可操作 &nbsp;&nbsp; ⚠️ = 条件限制

| 功能点 | system_admin | manager | user |
|--------|:---:|:---:|:---:|
| 登录系统 | ✅ | ✅ | ✅ |
| 查看 Dashboard | ✅ | ✅ | ✅ |
| 查看所有流程模板 | ✅ | ✅ | ✅ |
| 创建流程模板 | ❌ | ✅（本所） | ❌ |
| 编辑流程模板 | ❌ | ✅（本所） | ❌ |
| 删除草稿模板 | ❌ | ✅（本所） | ❌ |
| 发布流程 | ❌ | ✅（本所） | ❌ |
| 停用流程 | ❌ | ✅（本所） | ❌ |
| 发起流程实例 | ❌ | ✅（本所） | ❌ |
| 终止流程实例 | ❌ | ✅（本人发起） | ❌ |
| 执行节点（负责人） | ❌ | ✅ | ✅ |
| 上传文件 | ❌ | ✅ | ✅ |
| 提交节点 | ❌ | ✅ | ✅ |
| 审批任务（审批人） | ❌ | ✅ | ✅ |
| 查看流程实例 | ✅ | ✅ | ✅ |
| 查看操作日志 | ✅ | ✅ | ✅ |
| 用户管理 | ✅ | ❌ | ❌ |
| 组织管理 | ✅ | ❌ | ❌ |
| 角色分配 | ✅ | ❌ | ❌ |
| 重置密码 | ✅ | ❌ | ❌ |
| 系统配置 | ✅ | ❌ | ❌ |

## 2.3 权限规则

### 查看权限
- 所有已登录用户均可查看所有组织的流程模板
- 所有已登录用户均可查看所有流程实例
- 用户只能查看自己有权限访问的页面菜单

### 编辑权限
- 仅流程所属组织的所长可编辑该流程模板
- 草稿状态可编辑；已发布状态支持软修改（审批人/时限/描述等，即时生效不产生新版本）和硬修改（节点/连线变更，自动生成新版本）
- 其他组织人员仅可查看，不可编辑

### 执行权限
- 仅节点负责人可执行该节点（上传文件、提交）
- 仅配置的审批人可审批该节点
- 负责人与审批人允许跨组织

### 前端权限
- 前端根据角色隐藏无权限的按钮和菜单
- 真正权限校验统一由后端完成
- 任何 API 请求均需验证权限

# 3. 登录与认证

## 3.1 登录页面

### 页面元素
| 元素 | 说明 |
|------|------|
| 系统 Logo / 名称 | 居中展示"企业流程审批系统" |
| 用户名输入框 | 必填，支持字母数字下划线 |
| 密码输入框 | 必填，输入时遮蔽显示 |
| 登录按钮 | 点击后提交登录请求 |
| 错误提示 | 用户名或密码错误时红色提示 |

### 交互规则
1. 输入用户名和密码后点击"登录"
2. 系统校验用户名密码
3. 校验成功 → 返回 JWT Token，跳转至 Dashboard
4. 校验失败 → 显示"用户名或密码错误"，不清空输入框
5. Token 过期后自动跳回登录页
6. 未登录用户访问任何页面均跳转至登录页

### 状态处理
| 场景 | 表现 |
|------|------|
| 登录中 | 按钮显示 loading，禁止重复点击 |
| 登录成功 | 跳转 Dashboard |
| 登录失败 | 红色错误提示，保留输入内容 |
| Token 过期 | 自动退出，提示"登录已过期，请重新登录" |
| 账号被禁用 | 提示"账号已被禁用，请联系管理员" |

## 4.3 统计卡片

### 数据项

| 卡片 | 统计内容 | 说明 | 适用角色 |
|------|----------|------|----------|
| 进行中项目 | 全系统 Running 状态的实例总数 | 系统活跃度 | 全部 |
| 已归档项目 | 全系统 `archive_status=archived` 的实例总数 | 累计成果 | 全部 |
| 本月归档 | 本月 `archived_at` 非空的实例数 | 团队吞吐量 | 全部 |
| 超期预警 | 全系统已逾期 + 即将逾期的 Task 总数 | 风险预警 | 全部 |

### 交互规则
- 全部为全局统计，所有角色看到的内容一致
- 数据在页面加载时通过 API 获取
- 所有卡片均为纯展示，不做角色区分

## 4.4 任务状态分布（饼图）

### 数据说明
展示全系统所有任务的当前状态分布，一眼看出整体工作负载结构。

### 图表配置
| 配置项 | 内容 |
|--------|------|
| 数据类型 | 全系统 Task 状态分布（不含开始/结束节点） |
| 图表类型 | 饼图（环形或标准饼图均可） |
| 标签 | 扇区旁标注状态名 + 数量 + 百分比 |

### 扇区与颜色
| 状态 | 颜色 | 说明 |
|------|------|------|
| Pending（待处理） | 橙色 | 负责人尚未开始处理 |
| Processing（处理中） | 蓝色 | 负责人已开始但未提交 |
| Waiting Check（待校验） | 青色 | 已提交，等待校验中 |
| Waiting Approval（待审批） | 紫色 | 已通过校验，等待审批中 |
| Completed（已完成） | 绿色 | 近 30 天内完成的 Task |

### 交互
- 鼠标悬停扇区显示详细数量
- 点击扇区跳转至「我的待办」或「我的审批」（对应筛选状态）

## 4.5 流程卡点追踪（表格）

### 数据说明
展示所有运行中流程实例的完整进度——全部工作节点以标签链展示，已完成/进行中/待开始一目了然。

### 列表字段

| 列 | 说明 |
|----|------|
| 流程实例名称 | 流程发起时填写的名称 |
| 流程模板 | 使用的模板名称 |
| 所属组织 | 流程归属的所 |
| 节点进度 | 全部工作节点标签链：✅已完成 → 🔵进行中 → ⚪待开始。节点名称在标签内，当前节点显示负责人。超出一行自动折行 |
| 已耗时 | 从发起至今的天数 |
| 状态 | 正常 / 即将逾期 / 已逾期 |

### 节点进度示例

```
XT2026-001产品设计  产品设计流程  通用所
[✅资料上传 张三] → [✅设计评审 李四] → [🔵毛坯图下发 王五] → [⚪全套图纸] → [⚪BOM下发]
已耗时 12天  🟢正常
```

> 只显示工作节点（不含开始/结束节点）。节点标签可折行，表格行高自适应。

### 排序与筛选
- 首序按优先级：紧急 → 高 → 普通 → 低（优先级更高的排前面）
- 同优先级内按：已逾期 → 即将逾期 → 正常排序（最紧急的排前面）
- 支持按所属组织筛选
- 逾期/即将逾期的行红色/橙色高亮

### 逾期判定规则
| 标签 | 条件 |
|------|------|
| 🔴 已逾期 | 当前节点超过完成时限 |
| 🟡 即将逾期 | 距离完成时限 ≤ 2 天 |
| 🟢 正常 | 距离完成时限 > 2 天 |

## 4.6 超期预警（列表）

### 数据说明
展示全系统中已逾期和即将逾期的任务，帮助及时发现风险。

### 列表字段

| 列 | 说明 |
|----|------|
| 流程实例名称 | 所属流程 |
| 当前节点 | 逾期节点名称 |
| 负责人 | 节点负责人 |
| 截止时间 | 节点完成时限 |
| 剩余天数 | 还剩 X 天 / 🔴已逾期 X 天 |
| 所属组织 | 归属的所 |

### 排序
- 已逾期任务排最前面（逾期天数从多到少）
- 然后按剩余天数从少到多

### 数据说明
按组织分组，展示每个所下正在运行的流程实例。与流程管理模块的实例 Tab 对应。

### 交互

- 点击组织名折叠/展开

## 4.8 Dashboard 数据刷新

- 支持手动刷新按钮（右上角）
- 不做自动轮询（第一版）

# 5. 流程管理

## 5.1 模块概述

流程管理是所长创建和维护流程模板的入口。所有用户均可查看流程模板，但仅所长可在本所下编辑。

## 5.2 所概览卡片页

### 页面说明
进入流程管理时，以卡片形式展示所有组织概览，用户点击卡片进入该组织的流程列表。卡片数量随组织增减自动变化。

### 卡片信息

| 字段 | 说明 |
|------|------|
| 组织名称 | 如"通用所" |
| 运行中实例数 | 该所下 Running 状态的实例总数 |
| 最近更新时间 | 该所流程模板/实例最近一次更新时间 |

- 当前用户所属组织卡片高亮标记

### 全部流程卡片
在所有组织卡片之前，新增一张「全部流程」卡片：
- 卡片标题：🌐 全部流程
- 统计数：全系统实例总数
- hover 说明：查看所有组织的全部流程实例
- 点击进入全局实例列表（不限定组织）

### 交互规则
- 点击「全部流程」卡片进入全局实例列表，查看所有组织的全部流程实例
- 点击组织卡片进入该组织的流程列表
- 卡片数据从 API 动态获取，组织增减时自动反映

## 5.3 实例列表：流程实例 + 流程模板

### 页面概述
进入全部流程或某组织后，以 Tab 切换展示该范围的流程实例和流程模板。默认选中「流程实例」，因为查看运行中的流程是日常高频操作。全局视图（全部流程）下列表多一列「所属组织」，单所视图可隐藏该列。

### 页面布局

│  ┌──────────────────────────────────────────────────────────┐│

### 流程实例 Tab（默认选中）

#### 列表字段

| 列 | 说明 | 为什么需要 |
|----|------|-----------|
| 实例名称 | 发起时填写的名称 | 核心标识 |
| 模板来源 | 使用哪个流程模板 | 知道流程类型 |
| 优先级 | 🔴紧急 / 🟠高 / 🔵普通 / ⚪低（彩色标签） | 一眼看出轻重缓急 |
| 所属组织 | 实例所属的组织名称 | 全部流程视图显示，单所视图可隐藏 |
| 当前负责人 | 当前节点负责人，已完成则为"-" | 想催进度知道找谁 |
| 节点进度 | 当前第 N/M 个节点 | 一眼知道走到哪里了 |
| 状态 | 运行中 / 已完成 / 已终止 | 彩色标签区分 |
| 发起时间 | 实例创建时间 | 排序用 |

> 发起人信息在详情页查看，列表不展示。

#### 状态筛选
- 顶部状态筛选标签：全部 / 运行中 / 已完成 / 已终止
- 默认显示「运行中」

#### 操作
- 点击实例行进入该实例详情页（详见 §8 流程实例）

### 流程模板 Tab

该 Tab 下展示模板列表，所长可创建/编辑/管理模板。具体内容沿用 §5.4-§5.8（创建、编辑、发布、停用、删除、详情页、发起实例均保留）。

#### 列表字段

| 列 | 说明 |
|----|------|
| 流程名称 | 流程模板名称 |
| 当前版本 | 最新发布的版本号（如 V3） |
| 状态 | 草稿 / 已发布 / 已停用 |
| 节点数 | 模板中包含的节点数量 |
| 更新时间 | 最后修改时间 |
| 操作 | 查看 / 编辑 / 发布 / 停用 / 删除 / 发起（视状态与角色） |

#### 操作按钮可见性

| 操作 | 草稿 | 已发布 | 已停用 | 权限条件 |

## 5.5 编辑流程模板

### 修改类型

已发布模板的修改分为两类，避免小改也产生新版本：

| 修改类型 | 内容 | 是否产生新版本 | 生效方式 |
|----------|------|:--:|------|
| **软修改** | 仅 `description/assignee/time_limit/checkers/approvers` | ❌ 不产生 | 写入当前版本节点软配置覆盖层，仅影响之后发起的新实例；已发起实例完全快照隔离 |
| **硬修改** | 增删节点、改动连线、名称/文件要求/`is_optional` 等结构配置 | ✅ 自动版本号 +1 | 生成新版本，运行中实例不受影响 |

### 规则
- 草稿状态：自由编辑，不限软/硬修改
- 已发布状态：软修改直接保存生效，硬修改自动创建新版本
- 编辑不影响已运行中的流程实例（实例使用发起时的配置快照）
- 硬修改生成的新版本发布后，新发起的实例使用新版本

### 操作入口

- 硬修改：模板详情页 → 点击「编辑流程」→ 进入设计器修改结构 → 保存自动生成新版本

### 发布校验

发布前系统必须校验以下内容，任一不满足则阻止发布并提示：

| 校验项 | 规则 |
|--------|------|
| 流程名称 | 必填，2-50 字符 |

# 6. 流程设计器（Flow Designer）

## 6.1 模块定位

流程设计器是本系统核心功能模块，提供可视化界面供所长设计流程结构。

## 6.2 设计原则

系统采用**统一节点模型**：所有节点均为同一种类型（Work Node），不区分开始/工作/分支/结束。开始和结束只是在流程中位置特殊的节点。

V1 支持两种流程结构：

**串联（顺序执行）**：

```
┌──────┐     ┌──────┐     ┌──────────┐     ┌──────────┐     ┌──────────┐     ┌──────────┐     ┌──────┐
│ 开始  │ ──→ │资料上传│ ──→ │设计评审资料│ ──→ │毛坯图下发│ ──→ │全套图纸下发│ ──→ │整机BOM下发│ ──→ │ 结束  │
│(发起人)│     │      │     │          │     │          │     │          │     │资料下发   │     │(终审) │
└──────┘     └──────┘     └──────────┘     └──────────┘     └──────────┘     └──────────┘     └──────┘
  系统默认     工作节点      工作节点        工作节点        工作节点        工作节点        系统默认
```

**并行（分叉/汇合）**：

```
                 ┌──────┐
                 │现场  │
             ┌─→ │实施  │ ──┐
             │   └──────┘   │
┌──────┐ ┌──┴──┐           ┌↓──────┐ ┌──────┐
│ 开始  │→│概念  │           │项目  │→│ 结束  │
│(发起人)│ │设计  │           │验收  │ │(终审) │
└──────┘ └──┬──┘           └↑──────┘ └──────┘
             │   ┌──────┐   │
             └─→ │资料  │ ──┘
                 │整理  │
                 └──────┘
```

- **第一个节点（开始）**：系统默认生成，显示发起人姓名，无需配置
- **中间节点**：所长配置负责人、审批人、时限等信息。支持并行（一个节点完成后同时激活多个下游节点，全部完成后再汇合）
- **最后一个节点（结束）**：系统默认生成，审批人=发起人（终审），无需额外配置

## 6.4 节点说明

### 统一节点

所有节点使用相同矩形样式。节点的行为由其位置决定：

| 位置 | 显示 | 是否可配置 | 说明 |
|------|------|:--:|------|
| 第一个 | "开始" + 发起人姓名 | ❌ 不可配置 | 系统默认，新建流程时自动生成 |
| 中间 | 节点名称 + 负责人 + 时限 + 审批人 | ✅ 可配置 | 画布上直接显示关键配置信息 |
| 最后一个 | "结束（终审）" | ❌ 不可配置 | 系统默认，新建流程时自动生成 |

### 开始节点的行为
- 发起人点击"发起流程"后，自动跳过开始节点
- 流程直接进入第二个节点（第一个工作节点）
- 开始节点不生成 Task，不需要审批

### 结束节点的行为
- 最后一个中间节点审批通过后，流程自动进入结束节点
- 结束节点审批人 = 流程发起人（所长）
- 发起人在「我的审批」中看到终审任务
- 终审通过 → 流程结束 → 文件归档到服务器
- 终审驳回 → 选择目标节点 → 返回重新执行

### 节点视觉状态

#### 正常节点（配置完整）
```
┌──────────────┐
│ 📝 资料上传  │
│ 👤 李四     │
│ 🕐 7天      │
│ ✅ 王五,赵六│
└──────────────┘
```
- 实线边框，正常样式
- 四行信息：名称 / 负责人 / 时限 / 审批人
- 校验人和审批人每行最多显示 2 人，超出显示"+N"（如"王五,赵六 +1"）
- 鼠标悬停弹出 Tooltip 显示完整信息

#### 未配置节点（缺必填字段）
```
╭ - - - - - - - ╮
│ ⚠ 新节点  🔴 │
│ 👤 未配置   │
│ 🕐 未设置   │
╰ - - - - - - - ╯
```
- 橙色虚线边框，右上角红色圆点标记
- 必填字段未配置时显示"未配置"/"未设置"
- 配置完整后自动切换为正常样式
- 发布校验时这些节点也会在错误列表中列出

## 6.5 画布操作

### 添加节点
- 点击左侧"添加节点"按钮，自动在画布上新增一个工作节点
- 或从节点库拖拽至画布
- 新增的节点默认名称为"新节点"，需在属性面板中修改

### 删除节点
- 选中节点后按 Delete 键或右键菜单选择"删除"
- 第一个和最后一个节点不可删除（系统默认）
- 删除节点同时移除所有相关连线

### 连线操作
- 从节点输出端口拖拽至目标节点输入端口
- 点击连线可选中，按 Delete 删除
- 不支持自循环连线
- 第一个节点不可作为连线目标
- 最后一个节点不可作为连线源
- **分叉（fork）**：一个节点的输出可连接到多个下游节点，这些节点将并行执行
- **汇合（join）**：多个节点的输出可汇聚到同一个下游节点，该节点等待所有上游完成后才激活

### 画布缩放与平移
- 鼠标滚轮缩放（10% ~ 200%）
- 缩放按钮（+ / - / 适应画布）
- 拖拽画布空白区域平移

### 撤销与重做
- **Ctrl+Z** 撤销上一步操作
- **Ctrl+Y**（或 Ctrl+Shift+Z）重做被撤销的操作
- 工具栏提供撤销/重做按钮（显示在画布顶部工具栏）
- 支持撤销的操作：添加节点、删除节点、连线、断开连线、修改节点位置
- 撤销上限 50 步，超过后清除最早记录
- 属性面板中的字段修改由表单自身的撤销机制处理，不进入画布撤销栈

## 6.6 节点属性配置

选中中间节点后，在属性面板中配置：

| 属性 | 类型 | 必填 | 说明 |
|------|------|:--:|------|
| 节点名称 | 文本输入 | ✅ | 如"资料上传"，2-30 字符 |
| 节点描述 | 文本域 | ❌ | 描述本节点工作内容，最多 500 字符 |
| 负责人 | 用户搜索选择 | ✅ | 单选，支持跨组织 |
| 校验人 | 用户多选 | ✅ | 至少 1 人，支持跨组织。校验在审批前，全部通过后进入审批 |
| 完成时限 | 数字输入 | ✅ | 单位：天，正整数 |
| 必须上传文件 | 开关 | ❌ | 开启后，提交节点前必须上传至少一个文件 |
| 审批人 | 用户多选 | ✅ | 至少 1 人，支持跨组织 |
| 审批策略 | 单选 | 锁定 | 第一版固定"全部通过"，不可修改（预留扩展） |
| 可选节点 | 开关 | ❌ | 默认关闭。开启后发起实例时可选择跳过此节点 |

> **注意**：第一个和最后一个节点不可配置。选中时面板显示"系统默认节点，无需配置"。
>
> **负责人与审批人可为同一人**：系统不阻止负责人同时担任审批人，由各所在设计流程时自行决定。校验人和审批人也可重叠，但校验人和负责人不建议为同一人（自己校验自己失去意义，但系统不强制拦截）。
>
> **并行节点说明**：中间节点支持并行排列（分叉/汇合）。分叉后的各分支节点独立配置（各自有负责人、校验人、审批人、时限），互不影响。汇合节点（有多个上游连线的节点）需等待所有上游分支全部完成后才激活。

## 6.8 保存与发布

### 保存草稿
- 不进行完整性校验
- 保存当前设计器全部内容（节点 + 连线 + 属性）
- 提示"草稿保存成功"

### 发布
- 执行完整校验：
  - 流程名称：必填，2-50 字符
  - 节点数量：至少包含 3 个节点（开始 + 至少1个工作节点 + 结束）
  - 节点名称：每个工作节点名称必填
  - 负责人：每个工作节点必须指定负责人
  - 校验人：每个工作节点至少指定一个校验人
  - 审批人：每个工作节点至少指定一个审批人
  - 时限：每个工作节点必须设置完成时限
  - 连线：所有节点必须连通（无孤立节点）。分叉/汇合连线合法
- 校验通过 → 生成版本，状态变为 Published
- 校验失败 → 列出失败项，高亮对应节点
- 提示"流程发布成功，当前版本 V{n}"

   - Word/Excel → 通过 LibreOffice 无头模式转换
   - 图片 → 嵌入为 PDF 页面
   - 已是 PDF → 跳过转换

- Task 状态保持 Processing
- 可多次保存草稿

- 校验通过后，系统执行：

  2. 自动为每个校验人创建 CheckRecord 记录
  3. Task 状态变为 Waiting Check
  4. 记录操作日志
- 返回「我的待办」列表，提示"节点提交成功，文件已自动转换为PDF"

### 校验操作

#### 通过
1. 填写校验意见（可选）
2. 点击"通过"
3. 该校验人 CheckRecord 状态变为 Passed
4. 若所有校验人均通过 → 节点进入审批环节，自动为每个审批人创建 Approval 记录
5. 提示"校验通过，已进入审批环节"
6. 跳转回「我的待办」列表

#### 不通过（退回）
1. 必须填写校验意见（不通过原因）

4. Task 状态回到 Processing（负责人重新修改）

8. 跳转回「我的待办」列表

3. 该审批人 Approval 状态变为 Approved

   - 若是结束节点 → 流程状态变为 Completed，文件归档至服务器指定目录

#### 结束节点终审（发起人专属）
结束节点不同于普通节点，发起人直接进行终审，无"负责人提交"步骤：

1. 发起人在「我的审批」中看到终审任务（所属流程的所有中间节点均已通过）
2. 进入审批页面，可查看该流程**所有节点**的全部文件
3. 填写审批意见（可选）
4. 操作：
   - **通过** → 主状态变为 Completed，同时 `archive_status=archived` 并写入 `completed_at/archived_at` → 签名上 PDF
   - **总驳回** → 必须填写原因 → 从任意已执行中间工作节点中选择目标 → 返回该节点重新执行
5. 提示对应结果，跳转回「我的审批」

#### 审批意见
- 通过时可选填
- 驳回时必填
- 最多 500 字符
- 提交后不可修改

#### 驳回后文件处理规则

三种退回按以下文件规则处理：校验退回和中间审批退回均固定当前负责人并立即删除当前轮文件；终审总驳回删除目标节点文件，受影响下游在重新执行到达时删除旧文件：

1. 驳回节点的文件物理删除
2. 数据库文件记录同步删除
3. 负责人重新提交时上传新文件
4. 流程详情页展示节点当前文件
5. 被驳回节点的文件物理删除，已通过的审批签名随 PDF 一起删除

#### 驳回后下游节点级联规则

驳回目标节点之后的所有已完成节点，状态自动回退为 Waiting。重新轮到该节点执行时，旧文件删除、上传新文件。若流程在回退期间被终止，终止规则仍会删除实例全部物理文件和 files 记录。

示例（串联）：流程 `开始 → 节点2 → 节点3 → 节点4 → 结束`，节点4 驳回至节点2：
- 节点2 重新执行（生成新 Task，旧文件删除）
- 节点3、节点4状态回退为 Waiting；各节点重新执行到达时才删除其旧文件

#### 并行流程驳回规则

当流程包含 fork/join 结构时，驳回行为按驳回节点所在位置区分：

**场景一：驳回在分支内**

```
                ┌── 分支A ──┐
开始 → 分叉点 → ┤            ├→ 汇合点 → 结束
                └── 分支B ──┘
                    ↑
                 驳回至此
```

- 仅分支 B 及其下游到汇合点之间的节点受影响
- 分支 A 不受影响，继续正常运行
- 汇合点需等待两个分支全部完成，分支 B 重跑完成后汇合点恢复等待

**场景二：驳回在分叉点或分叉点之前**

```
开始 → 节点2 → 分叉点 →  ┌── 分支A ──┐
     ↑                    └── 分支B ──┘
  驳回至此
```

- 分叉点之后的所有分支（A 和 B）全部回退
- 所有分支节点状态回退为 Waiting，重新执行
- 原分支文件的处理规则与串联流程一致

#### 签名规则
- 每个用户需预先在个人信息中上传签名图片（PNG 格式，透明底，建议 200×60 像素）
- 审批通过时，系统自动使用 `pypdf` 库将签名图片插入 PDF 指定坐标
- 签名位置由系统配置统一设定（默认：PDF 最后一页右下角，起始坐标 X=400, Y=100）
- 多审批人签名按审批顺序水平排列，每个签名向右偏移 150px（可在系统配置中调整），超出页面宽度自动换行
- 多签名不互相覆盖
- 未上传签名的用户可正常审批，审批通过时跳过签名插入步骤

| 概要栏 | 状态图标 + 节点名称 + 负责人 + 时限/截止时间 | 折叠时也可见 |

| 🔍 校验 | 校验记录列表（校验人、状态、意见、时间） | 无校验人时隐藏此区域 |
| ✅ 审批 | 审批记录列表（审批人、状态、意见、时间、签名状态） | 无审批记录时隐藏此区域 |
| 📋 操作记录 | 本节点相关的操作日志 | 仅展示与本节点直接相关的操作 |

## 8.4 操作日志

### 数据结构
按时间倒序展示该流程实例的全部操作日志。

### 每条记录
| 字段 | 说明 |
|------|------|
| 时间 | 操作时间 |
| 操作类型 | 发起 / 进入节点 / 上传文件 / 提交 / 审批通过 / 审批驳回 / 完成等 |
| 轮次 | 第几次执行该节点（1 开始，驳回重新进入后 +1） |
| 描述 | 自动生成的描述文本 |
| 操作人 | 操作者姓名 |

### 规则
- 分页或滚动加载
- 默认显示最近 50 条
- 不可删除
- 日志表按年分区存储，长期保留可随时查询，无需人工清理
- 每条记录显示所属轮次，方便区分同一节点的多次执行

## 8.5 文件补交（兜底机制）

### 页面说明
除 Terminated 外可按状态和身份补交文件。Running 时用于补充当前节点材料，Completed（无论是否已归档）时用于补充对应历史节点材料。补交不触发审批/校验，不影响流程状态和节点推进。

### 入口

| 入口 | 位置 | 可用阶段 |
|------|------|----------|
| 任务处理页 | 附件区「补交文件」按钮 | 节点执行中（负责人可见） |

| 详情页节点卡片 | 每个节点卡片文件区「补交文件」 | 任意阶段 |
| 详情页顶部 | 右上角全局「补交文件」 | 任意阶段 |

### 权限
- **进行中节点**：流程发起人 + 当前节点负责人
- **Completed（无论是否已归档）**：流程发起人 + 对应节点历史负责人；校验人/审批人身份本身不授权
- **Terminated**：禁止补交

### 补交流程
1. 点击「补交文件」
2. 选择本地文件上传（支持多文件，单文件 ≤50MB）
3. 文件自动转换为 PDF
4. 存储至 `{实例名称}/` 目录（同正常文件）
5. 数据库 files 记录：`node_id = 当前节点ID`, `upload_type = 'supplement'`
6. 记录操作日志

### 展示
- 各节点卡片文件区展示该节点的补交文件，与正常文件分开展示，标注「补交」标签
- 补交文件显示补交人和补交时间
- 支持下载

## 8.6 修改节点人员（紧急换人）

### 页面说明
流程运行中，若有人员临时请假等意外情况，发起人可随时更换未完成节点的负责人、校验人或审批人，确保流程不卡住。此功能为兜底机制，仅在必要时使用。

### 入口
流程实例详情页 → 节点卡片展开 → 「修改人员」按钮（仅发起人可见）。

### 权限
- 仅流程发起人（所长）可操作
- 仅可修改未完成状态的节点（waiting / running / waiting_check / waiting_approval）

### 修改弹窗

```
┌── 修改节点人员 ────────────────────────────────────────┐
│                                                         │
│  节点：设计评审资料                                       │
│  状态：🔄 运行中                                         │
│                                                         │
│  负责人：[赵六            ▼]  ← 可更换                   │
│                                                         │
│  校验人：                                               │
│    [钱七 ×] [孙八 ×] [+ 添加]  ← 可增删                  │
│                                                         │
│  审批人：                                               │
│    [李四 ×] [王五 ×] [+ 添加]  ← 可增删                  │
│                                                         │
│  ⚠️ 提示：                                              │
│  - 当前校验人已有 1 人通过，不会受到影响                  │
│  - 仅替换尚未操作的 pending 校验人和审批人                │
│  - 变更将记录操作日志                                    │
│                                                         │
│              [取消]         [确认修改]                    │
└─────────────────────────────────────────────────────────┘
```

### 后端处理
1. 更新 `instance_nodes` 中的 assignee_id / checkers / approvers
2. 遍历该节点已有的 CheckRecord：
   - 若校验人不在新列表中 → 无影响（旧记录保留）
   - 若校验人仍在列表中且记录为 pending → 保留
   - 新加入的校验人 → 生成新的 CheckRecord（status=pending）
3. 遍历该节点已有的 Approval：
   - 若审批人不在新列表中 → 无影响（旧记录保留，pending 记录关闭为 terminated）
   - 若审批人仍在列表中且记录为 pending → 保留
   - 新加入的审批人 → 生成新的 Approval（status=pending）
4. 记录操作日志（变更详情：哪个字段，从谁变成谁）

### 业务规则
- 已完成节点不可修改（状态为 finished / skipped 的节点）
- 已终止流程不可修改
- 不限制修改次数
- 每次修改独立记录操作日志
- **仅发起人可修改 Running 实例的优先级**，在实例详情页顶部基本信息区直接修改，操作记入日志

# 附录 B：状态流转图

## 流程模板状态
```
                    ┌── 软修改（即时生效，不产生新版本）──┐
                    ↓                                   │
Draft ──发布──→ Published ──硬修改（节点/连线变更）──→ Draft（新版本）
                 │
                 └──停用──→ Disabled
```

## 流程实例状态
```
Created ──→ Running ──→ Completed；archive_status 同步变为 Archived（V1）
  │            ↑    │
  │            │    ├──→ Terminated
  │            │    │    （发起人终止，文件删除，不可恢复）
  │            │    │
  │            │    └── 终审驳回（返回指定节点重新执行）
  │            │
  └── 终止 ────┘
```

> **Created 状态说明**：Created 是瞬时内部状态——实例创建后立刻推进到 Running，用户不应看到此状态。保留它仅为保护系统异常时的数据完整性。

> **归档说明**：主状态不含 Archived。V1 终审通过时主状态置 Completed，同时将 `archive_status` 置 Archived 并写入 `archived_at`。文件本就存储在 `storage/archive/{实例名称}/` 下，不涉及物理移动。

## 节点状态（统一模型）

中间工作节点：
```
Waiting ──→ Running ──→ Waiting Check ──→ Waiting Approval ──→ Finished
                          ↓                ↓
                   Running（校验/中间审批退回当前负责人）  Rejected（仅终审总驳回跨节点重跑）
```

开始节点（首个节点）：
```
Waiting ──→ Finished（发起流程后自动完成）
```

结束节点（最后一个节点）：
```
Waiting ──→ Waiting Approval（发起人终审）──→ Finished
              ↓
           Rejected（驳回至指定节点）
```

## Task 状态
```
Pending ──→ Processing ──→ Waiting Check ──→ Waiting Approval ──→ Completed
                              ↓                ↓
                     Processing（校验/中间审批退回）  Rejected（终审总驳回关闭旧 Task）
```

> 注意：开始节点和结束节点不生成 Task。开始节点自动跳过；结束节点发起人直接在「我的审批」中终审。

## CheckRecord 状态
```
Pending ──→ Passed（进入审批）
Pending ──→ Returned（退回负责人修改）
Pending ──→ Terminated（流程终止/紧急换人）
```

## Approval 状态
```
Pending ──→ Approved（签名自动上 PDF）
Pending ──→ Rejected
```

## 完整流程状态流转
```
发起流程
  ↓
开始节点（自动跳过）
  ↓
工作节点1：负责人执行 → 上传 → 转PDF → 提交 → 审批（签名上PDF）→ 进入下一节点
  ↓
工作节点2：负责人执行 → 上传 → 转PDF → 提交 → 审批（签名上PDF）→ 进入下一节点
  ↓
...
  ↓
结束节点：发起人终审
  ├── 通过 → 归档至服务器 → Completed
  └── 驳回 → 返回指定节点 → 重新执行
